Introducing

Ember.js

The OMG of it.
Ember.js is an MVC framework for building client-rich applications*

* also known as single-page apps

Designed similarly to Ruby on Rails where convention outweighs configuration thus less code is written that's more meaningful.
Features data binding, integrated templating and works on older browsers too!*

*IE7 and above

Lets get started

It all starts with Ember.Object

var person = Ember.Object.create({
    firstName: 'Chuck',
    lastName: 'Testa',

    fullName: function() {
        return [this.get('firstName'),
          this.get('lastName')].join(' ');
    }.property('firstName', 'lastName')
});

Data bindings, like quantum coupling but in Javascript

App.red = Ember.Object.create({
  color: 'red'
});
App.blue = Ember.Object.create({
  colorBinding: 'red.color'
});
App.blue.get('color'); // red

Templates make outputting easy and are designed to contain little logic.

{{#each person in people}}
  Hello {{person.fullName}}
{{/each}}

Complex presentation logic are handled by Views.

var view = Ember.View.create({
  template: 'people',
  click: function() {
    alert('clicked!');
  }
});

Along to controllers, routes are used to control application state.

They behave similarly to request-based frameworks.

App.Router.map(function() {
  this.resource('people', function() {
    this.route('show', {path: '/:person_id'});
  });
});

App.PeopleRoute = Ember.Route.extend({
  model: function(params) {
    return [];
  },
  setupController: function(controller, people) {
    controller.set('people', people);
  }
});

Models on the client-side are tricky and currently there's no silver bullet.

Backends implement non-standard data structure and APIs, making it very difficult.

Ember Data is trying to solve this problem in a conventional way.

Don't like it?
Use a different implementation.
There are dozens already!

What are the benefits?

  • Rich and complex UIs
  • More responsive interactions
  • Clean separation of front and back end
